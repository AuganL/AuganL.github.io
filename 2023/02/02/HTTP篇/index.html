什么是HTTP

  • http是计算机世界里的协议，确立了一种计算机之间交流通信的规范以及相关的控制和错误处理方式
  • http专门用于在两点之间传输数据
  • http不能用于广播、寻址、路由
  • http传输的是文字、图片、音频、视频等超文本数据
  • http是构建互联网的重要集成技术
  • http没有实体，http只是一种在计算机世界里规定的一种规范

  • 互联网上绝大部分资源都使用http协议传输

  • 浏览器是http协议里的请求方，即请求头中的User-Agent属性就会对请求方进行记录

  • 服务器是http协议里的应答方，常用的有Apache和Nginx

  • CDN位于浏览器和服务器之间，主要起到缓存加速的作用

  • 网络爬虫对应的是另一类User-Agent，是自动访问网络资源的程序

  • TPC/IP是网络世界最常用的协议，http通常运行在TCP/IP提供的可靠传输基础上
  • DNS域名是IP地址的等价替代，需要用域名解析实现从域名到IP地址的映射
  • URL是用来标记互联网上资源的一个名字，由“协议名+主机名+路径”构成
  • HTTPS相当于HTTP+SSL/TLS+TCP/IP,为http套了一个安全的外壳
  • 代理是HTTP传输过程中的“中转站”，可以实现缓存加速、负荷均衡等功能

OSI网络七层模型以及TCP/IP

OSI网络模型分为七层自下而上为

  • 物理层
  • 数据链路层
  • 网络层
  • 传输层
  • 会话层
  • 表示层
  • 应用层

TCP/IP分为四层

  • 第一层与OSI七层模型无对应
  • 第二层链路层，对应OSI七层模型的数据链路层
  • 第三层网络层，对应OSI七层模型的网络层
  • 第四层传输层，对应OSI七层模型的传输层
  • 最上层应用层与OSI七层模型的第五、六、七层对应

http利用TCP/IP协议栈逐层打包再拆包，实现了数据传输

除了80端口，http协议还经常所以8000和8080端口

http报文

报文结构

http协议的报文分为请求报文和响应报文。但是结构基本都相同

http报文结构，由三大部分组成：起始行、头部字段集合、消息正文

  • 起始行：描述请求或响应的基本信息
  • 头部字段集合：使用key-value（键值对）的形式详细的说明报文
  • 消息正文：实际传输的数据，不一定是纯文本，可以是其他像图片、视频一系列的二进制数据
  • 起始行和头部字段集合经常合称为“请求头”或者“响应头”，消息正文又称为实体
  • http协议规定报文必须有header，但可以没有body
  • header 和 body 之间必须有一个“空行”

http协议与TCP/IP类似，同样需要在实际传输的数据前附加一些头数据，不过与TCP/IP不同的是http是一个纯文本协议，所以头数都是ASCII码的文本，不用借助程序解析也可以看懂。

请求行

简要的描述了客户端想要如何操作服务端的资源

请求行的构成

  • 请求方法：如GET/POST
  • 请求目标：通常是一个URL，标记了请求方法要操作的资源
  • 版本号：表示报文使用的HTTP协议版本
  • 这三个部分通常使用空格（space）来分隔，最后要用 CRLF 换行表示结束。

状态行

状态行的构成

  • 版本号：表示报文使用的http协议版本

  • 状态码：表示请求处理结果

  • 原因：作为状态码的补充，描述返回该状态码原因

头部字段集合

请求行或者状态行再加上头部字段集合就构成了HTTP报文里完整的请求头或者响应头

  • 请求头和响应头的结构基本是一样的，唯一的区别是起始行不同

头部字段是 key-value 的形式，key 和 value 之间用“:”分隔，最后用 CRLF 换行表示字段结束。比如在“Host: 127.0.0.1”这一行里 key 就是“Host”，value 就是“127.0.0.1”。

HTTP 头字段非常灵活，不仅可以使用标准里的 Host、Connection 等已有头，也可以任意添加自定义头，这就给 HTTP 协议带来了无限的扩展可能。

http请求方法

HTTP规定了八种请求方法

  • GET：获取资源，读取或者下载数据
  • HEAD：获取资源的元信息
  • POST：向资源提交数据，相当于写入或者上传数据
  • PUT：类似POST
  • DELETE:删除资源
  • CONNECT：建立特殊的连接隧道
  • OPTIONS：列出可对资源实行的方法
  • TRACE：追踪请求 - 响应 的传输路径

URI

  • URI是用来唯一标记服务器上资源的一个字符串，通常也称为URL
  • URI通常由scheme、host：port、path和query四个部分组成，有的可以省略
  • scheme叫协议名，表示资源应该使用哪种协议来访问
  • host：port表示资源所在的主机名和端口号
  • path标记资源所在的位置
  • query表示对资源附加的额外要求

HTTP状态码

  • 1xx 信息性状态码 websocket upgrade
  • 2xx 成功状态码
    • 200 服务器已成功处理了请求
    • 204(没有响应体)
    • 206(范围请求 暂停继续下载)
  • 3xx 重定向状态码
    • 301(永久) ：请求的页面已永久跳转到新的url
    • 302(临时) ：允许各种各样的重定向，一般情况下都会实现为到 GET 的重定向，但是不能确保 POST 会重定向为 POST
    • 303 只允许任意请求到 GET 的重定向
    • 304 未修改：自从上次请求后，请求的网页未修改过
    • 307：307 和 302 一样，除了不允许 POST 到 GET 的重定向
  • 4xx 客户端错误状态码
    • 400 客户端参数错误
    • 401 没有登录
    • 403 登录了没权限 比如管理系统
    • 404 页面不存在
    • 405 禁用请求中指定的方法
  • 5xx 服务端错误状态码
    • 500 服务器错误：服务器内部错误，无法完成请求
    • 502 错误网关：服务器作为网关或代理出现错误
    • 503 服务不可用：服务器目前无法使用
    • 504 网关超时：网关或代理服务器，未及时获取请求

http的特点

  • HTTP 是灵活可扩展的，可以任意添加头字段实现任意功能；
  • HTTP 是可靠传输协议，基于 TCP/IP 协议“尽量”保证数据的送达；
  • HTTP 是应用层协议，比 FTP、SSH 等更通用功能更多，能够传输任意数据；
  • HTTP 使用了请求 - 应答模式，客户端主动发起请求，服务器被动回复请求；
  • HTTP 本质上是无状态的，每个请求都是互相独立、毫无关联的，协议不要求客户端或服务器记录请求相关的信息

http的优缺点

  • HTTP 最大的优点是简单、灵活和易于扩展；
  • HTTP 拥有成熟的软硬件环境，应用的非常广泛，是互联网的基础设施；
  • HTTP 是无状态的，可以轻松实现集群化，扩展性能，但有时也需要用 Cookie 技术来实现“有状态”；
  • HTTP 是明文传输，数据完全肉眼可见，能够方便地研究分析，但也容易被窃听；
  • HTTP 是不安全的，无法验证通信双方的身份，也不能判断报文是否被窜改；
  • HTTP 的性能不算差，但不完全适应现在的互联网，还有很大的提升空间

http的实体数据

数据类型和编码

  • 数据类型表示实体数据的内容是什么类型，使用的是 MIME type，相关的头字段是 Accept和 Content-Type
  • 数据编码表示实体数据的压缩方式，相关的头字段是 Accept-Encoding 和 Content-Encoding

数据类型

  • text ：表示文本格式的可读数据。text/html（超文本文档）、text/plain（纯文本）、text/css（样式表）
  • image：表示图像文件。image/gif、image/jpeg、image/png
  • audio/video：no安排上音频和视频数据。audio/mpeg、video/mp4
  • application：application/json、application/Javascript、application/pdf、application/octet-stream

数据编码

  • gzip：GNU zip 压缩格式，也是互联网上最流行的压缩格式；
  • deflate：zlib（deflate）压缩格式，流行程度仅次于 gzip；
  • br：一种专门为 HTTP 优化的新压缩算法（Brotli）

http传输大文件

数据压缩

通常浏览器在发起请求的时候，都会带上“Accept-Encoding”头字段，表示浏览器支持的压缩格式列表，例如gzip、deflate、br等。这样服务器就可以从其中选择一种压缩算法，然后将选择的压缩算法放进“Content-Enocoding”响应头里，再把原数据压缩后发给浏览器

  • 如果压缩率能有 50%，也就是说 100K 的数据能够压缩成 50K 的大小，那么就相当于在带宽不变的情况下网速提升了一倍，加速的效果是非常明显的。

分块传输

除了数据压缩之外，http还通过分块传输的方式来处理大文件传输问题。

这种“化整为零”的思路在 HTTP 协议里就是“chunked”分块传输编码，在响应报文里用头字段“Transfer-Encoding: chunked”来表示，意思是报文里的 body 部分不是一次性发过来的，而是分成了许多的块（chunk）逐个发送。

分块传输的编码规则

  • 每个分块包含两个部分，长度头和数据块；
  • 长度头是以 CRLF（回车换行，即\r\n）结尾的一行明文，用 16 进制数字表示长度；
  • 数据块紧跟在长度头后，最后也用 CRLF 结尾，但数据不包含 CRLF；
  • 最后用一个长度为 0 的块表示结束，即“0\r\n\r\n”

范围请求

  • http协议为了满足上G的超大文件传输，提出了范围请求的概念
  • 范围请求允许客户端在请求头里使用专用字段来表示只获取文件的一部分
  • 服务器必须在响应头里使用”Accept-Ranges：bytes”字段告知客户端，服务器是支持范围请求的

段数据

  • 刚才说的范围请求一次只获取一个片段，其实它还支持在 Range 头里使用多个“x-y”，一次性获取多个片段数据。
  • 这种情况需要使用一种特殊的 MIME 类型：“multipart/byteranges”，表示报文的 body 是由多段字节序列组成的，并且还要用一个参数“boundary=xxx”给出段之间的分隔标记。
  • 多段数据的格式与分块传输也比较类似，但它需要用分隔标记 boundary 来区分不同的片段，可以通过图来对比一下
